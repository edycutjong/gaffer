## Step 1 — Hard disqualifiers

  the same match topic — and the gauge surges to ~47 tok/s ... Kill the laptop mid-sentence
  and the sentence finishes anyway."* This is stated as an on-screen event, not a background
  process. **PASS.**
- **Needs a live external API on stage?** README's own compliance table: *"Hyperswarm
  discovery, Noise-encrypted delegation, token streaming ... Real, always — including in
  tests (loopback DHT testnet, no mocks)"* and a dedicated `npm run verify:offline` command is
  advertised as a live, on-camera step in the video script. No external API is named anywhere
  as required for the demo to work. **PASS.**
- **Only impresses at scale?** Docs describe a two-terminal / two-machine demo throughout
  (`docs/DEMO.md`); no claim depends on populated data or many users. **PASS.**
- **Core feature only works on canned input?** The docs disclose the honest limit themselves:
  *"Match events | Deterministic simulator (seeded fixtures) — Gaffer narrates feeds; it is
  not a licensed data product."* This is a scoped, disclosed limitation, not a hidden canned
  demo — and the offload/failover mechanism it's paired with (the actual judged capability)
  is not input-dependent. **PASS, with disclosure already in place** (nothing to fix).
- **One-sentence problem statable from docs alone?** README line 3: *"expert match
  commentary dies the moment the network drops — and the fans who most need an explainer
  carry the weakest phones."* **PASS**, and it's the literal first line after the title.

No disqualifier fires. Proceeding to scored tests.

## Step 2 — Scored tests (A–E)

### A. Shippable (2× buffer, honesty test) — **9/10**

The docs don't just assert "it's done" — `docs/gaffer_dorahacks_submission.md` §9 is an
explicit checklist that separates what's finished (everything code/test/CI) from what needs
the account owner (push, video, form fields), and flags each with **[user]**. Doubling the
estimate on the one open task — recording a clean two-terminal video with a correctly timed
Ctrl-C — is still an hour-scale task, not a days-scale one. The single task most likely to
eat any buffer is the video: it's the only artifact that isn't regenerated by a script, so a
bad take costs re-recording time.

### B. Winnable (judging-reality test) — **top third of ~53**

The docs pre-empt this test themselves in `docs/REVIEW_AUDIT.md`: *"Expected field
archetypes: chat-with-your-team RAG bots ... wallet dApps, prediction games. A weak phone
thinking with a laptop's brain, fully offline, is a different kind of demo."* That's a
credible self-placement for a knockout bracket of this size — a judge skimming 40+ READMEs
will not have seen "P2P inference offload with mid-stream failover" as a demo shape before.
The archetype most likely to out-shine it on raw spectacle is a team running the *real* model
live on stage rather than the disclosed sim engine — the docs already name this as the
highest-leverage next step, which is itself a good sign (self-aware roadmap, not overclaiming
readiness it doesn't have).

### C. Wow-factor & magic moment — **8/10**

Rated 0–10 on how hard the judge's eyebrow moves in 30 seconds: the README's own framing —
*"a weak client crawls at ~6 tok/s. A laptop joins the same match topic — and the gauge
surges to ~47 tok/s with a lime `⇄ BOOSTED BY PEER` badge. Kill the laptop mid-sentence and
the sentence finishes anyway, token-exact"* — is a strong, single-sentence magic moment
already written into the docs, not something this audit had to design.

**Magic-moment beat, timestamped** (from the video script in
`docs/gaffer_dorahacks_submission.md` §7): *"0:20–1:00 · The surge — HUD gauge 6→47 tok/s,
⇄ BOOSTED BY PEER"* and *"2:05–2:40 · The kill — Ctrl-C the provider mid-sentence; the
sentence finishes; state flips to FALLBACK."* Two "oh" beats in one 3-minute script.

Docked 2 points, not for the mechanism, but because the docs are honest that the tokens
themselves come from a disclosed sim grammar by default (`--engine qvac` is the real-model
path but not what the default judge run shows) — a sharp judge who reads the "what's real vs
simulated" table will register that the wow is in the transport, not (by default) the model.

### D. Non-generic — **8/10**

Named comparables the docs already volunteer (`docs/REVIEW_AUDIT.md` §D): *"local llama chat
app" (no P2P), "P2P chat over Hyperswarm" (no AI), "watch-party app" (cloud).* None combine
P2P inference offload with mid-stream failover, and the README's "Why ONLY QVAC + Pear"
section spells out, feature by feature, what a cloud-based clone would need instead (a cloud
LLM endpoint, a cloud TTS endpoint, a model CDN, a signalling/TURN service, a rented GPU) —
this is a strong non-genericness argument made in the judge's own reading path, not buried in
a design doc. Not a 9–10 because the underlying AI capability (a small GGUF model) is
off-the-shelf; novelty is concentrated in the delegation layer.

### E. Documentation polish — **9/10**

- **Opening hook:** README opens *Problem → Solution → Built* in the first 5 lines
  (title, then a `>` blockquote with exactly that structure) before any setup instructions —
  this is the strongest form of the pattern, not buried.
- **Submission gaps:** `docs/gaffer_dorahacks_submission.md` explicitly flags every
  account-gated field with **[user]** (repo URL, video URL, nation, team, location) rather
  than leaving them silently blank or faked — a judge reading the submission doc sees exactly
  what's missing and why (logistics, not capability).
- **Claims the demo can't back up:** none found. Every numeric claim (158 tests, ×7.8
  speedup, 97%+ transport efficiency, Apache-2.0) is paired with the exact command that
  reproduces it (`npm test`, `npm run bench`, `npm run verify:offline`), and
  `scripts/check_submission_readiness.js` re-verifies the README's stated test count against
  a **live** run before submission — this is a stronger claims-integrity mechanism than most
  hackathon docs attempt.
- **Polish nits:** none material found. No dead links to `OWNER/REPO`-style placeholders (the
  README carries no GitHub-hosted CI badge, sidestepping the "badge is red because repo isn't
  public yet" trap other projects fall into). One cosmetic overlap: an earlier self-audit
  (`docs/REVIEW_AUDIT.md`) and this document now both exist in `docs/`, which functions but
  isn't unified into one canonical trail.
